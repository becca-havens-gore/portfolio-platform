Case Study — Platform Interaction Architecture

Eliminating Context-Switching Friction Across a Fragmented Multi-Product Platform

How I designed a unified cross-surface interaction system that gave technical users a coherent mental model — regardless of which product they were working in.

The Problem: Every Product Spoke a Different Language

Across Rapid7's six security products, the fundamental interactions that practitioners performed dozens of times per day — filtering lists, sorting tables, selecting rows, triggering bulk actions — were implemented differently in every product. Not cosmetically differently. Behaviorally differently.

A security engineer moving between InsightIDR (SIEM/detection), InsightVM (vulnerability management), and InsightOps (log management) had to maintain separate mental models for what appeared to be the same interaction pattern. Keyboard shortcuts didn't transfer. Multi-select behavior was inconsistent. Filter chips worked differently. Bulk action menus appeared in different locations.

These patterns collectively account for 70–85% of all user actions in the platform. The cognitive tax was not occasional — it was continuous.

"Every product expects me to relearn how to do the same thing. Tables, filters, selections — nothing works the same way twice."

— Security engineer, cross-product research session

What Fragmentation Actually Cost

User Impact

  • Constant re-learning when switching products
  • Higher error rates on bulk operations
  • Keyboard users hit dead ends — focus traps, missing tab targets
  • Filter behavior mismatch caused incorrect queries
  • Increased time-on-task for routine triage workflows

Engineering & Design Cost

  • 6 separate DataGrid implementations being maintained in parallel
  • Designers re-solving the same interaction problems independently
  • No shared spec meant engineers guessed at missing behavior
  • Accessibility gaps discovered late and fixed inconsistently
  • High onboarding cost for new engineers joining platform teams

Why This Is a Developer Platform Problem

Developer platforms face the same coherence challenge at larger scale. When a developer moves between their IDE, a CI/CD dashboard, an observability tool, and a deployment interface — all within a single platform ecosystem — inconsistent interaction models create exactly the same cognitive overhead.

The UPX work maps directly to the cross-surface coherence challenges in integrated DevOps platforms (GitLab, GitHub, and similar): how do you design filtering, navigation, table interactions, and bulk operations that feel native everywhere, so engineers can focus on the work rather than the tool?

This is the core of "Paved Road" thinking: designing smart defaults and guided experiences that reduce cognitive load, so engineers can move quickly and safely without needing to be experts in every surface of the underlying infrastructure.

The patterns I designed — DataGrid interactions, filtering models, keyboard navigation, batch action architecture — are the foundational vocabulary of any technical platform where practitioners spend hours per day in dense data views.

Strategy: Normalize Behavior, Not Just Appearance

The most important framing decision in this project: UPX was an interaction system, not a visual redesign. The goal was behavioral coherence — making the same action work the same way everywhere — not making products look identical.

This distinction mattered for engineering buy-in. Teams didn't need to rebuild their UIs. They needed to adopt shared behavioral contracts for how core interactions worked.

1
DataGrid Interaction Framework
Unified behavioral spec for Rapid7's most-used UI surface. Covered hover states, row/cell focus, single and multi-select, expand/collapse, inline editing, right-click context menus, and complete keyboard navigation. Defined as a contract any DataGrid implementation must fulfill, regardless of underlying library.
2
Filtering & Query Model
Cross-product standard for how filtering works: horizontal filter bars, query chip creation and removal, clear-all behavior, time-range filter models, and filter state persistence across sessions. Unified the mental model that practitioners used to narrow data across every product view.
3
Batch Action Architecture
Standardized how bulk operations were triggered, confirmed, and executed. Resolved the most common source of user error: batch actions hidden in menus or triggered without adequate confirmation. Defined persistent action bars, progressive confirmation patterns, and error recovery for partial batch failures.
4
Keyboard Navigation & Accessibility Model
Complete keyboard interaction model for all DataGrid, filter, and batch action surfaces. Eliminated the focus traps, missing tab targets, and inaccessible selection flows that were blocking keyboard-dependent users from completing core workflows. Integrated WCAG 2.2 AA requirements directly into the interaction spec.

The Interaction Patterns That Mattered Most

Six interaction patterns had the highest impact on practitioner workflow efficiency. These were the areas where inconsistency was doing the most damage:

Multi-Select with Persistence
Unified how rows stay selected across sort, filter, and page changes. Previously, filtering a view deselected all rows — forcing users to re-select after narrowing. Selection state now persists until explicitly cleared.
✓ High impact for batch triage workflows
Query Chip Model
Standardized how applied filters are displayed, combined, and removed. Chips show the active filter state, support group-level removal, and expose the underlying query for power users. Consistent across all six products.
✓ Core to log/trace/metric filtering
Inline Editing with Conflict Resolution
Defined when inline editing is appropriate vs. modal editing, and how to surface and resolve edit conflicts when multiple users are working in the same data view. Eliminated the pattern of silently overwriting another user's change.
✓ Collaborative platform surfaces
Persistent Batch Action Bar
Replaced hidden dropdown bulk actions with a persistent contextual action bar that appears on multi-select. Actions stay visible, show count of selected items, and surface confirmation requirements before destructive operations.
✓ Eliminates the most common error pattern
Expandable Row with Context Preservation
Unified expand/collapse behavior so detail views open inline without losing list position. Users can scan context alongside detail — particularly important for investigation workflows where comparing list and detail simultaneously is essential.
✓ Observability drill-in analog
Time-Range Filter Model
Standardized the most complex filter type across the platform. Unified relative ranges (last 24h, last 7d), absolute date pickers, and custom range entry into one consistent UI that works identically in dashboards, tables, and reports.
✓ Essential for telemetry-heavy platforms

The Tradeoff We Accepted

What We Accepted

We did not require product teams to rebuild their DataGrid implementations simultaneously. Teams could adopt the UPX behavioral spec on their existing implementation, then migrate to a shared component library in a later phase. Some products had behavioral coherence before they had a unified component.

Why

Requiring a full component migration before adopting the spec would have pushed the user experience benefit 12–18 months into the future. Users benefited from behavioral consistency immediately, while engineering handled the technical unification on their own timeline. Experience coherence and implementation unification are separate problems — solving them together shouldn't be a prerequisite.

Making It Stick: Adoption Without Mandate

Cross-product design standards fail when they're treated as mandates handed down from a central team. I designed the adoption model as a service, not a requirement:

  • Implementation-ready specs: Every interaction was documented to the level of detail where an engineer could implement it without asking a designer a single question — states, transitions, keyboard mappings, ARIA patterns, edge cases.
  • "Ask the Platform" sessions: Weekly open office hours where product teams could bring implementation questions, edge cases, and exception requests. This created a feedback loop that improved the spec over time.
  • Phased rollout with wins first: We sequenced adoption so teams tackled the highest-visibility patterns first (DataGrid selection, filter chips) and saw user research validation before moving to more complex interactions.
  • Exception tracking: Teams could formally request exceptions with documented rationale. This prevented covert non-compliance and built a record of genuine edge cases that informed spec evolution.
